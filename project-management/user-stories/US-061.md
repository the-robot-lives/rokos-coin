# US-061: Use the Marketplace for Non-LLM Compute (Image Gen, Speech)

**As a** app developer/API consumer,
**I want to** use the ROKO marketplace for compute tasks beyond text generation — including image generation, speech-to-text, and embeddings,
**So that** I can build multimodal applications on a single decentralized compute platform without juggling multiple providers.

## Acceptance Criteria
- [ ] The model registry supports multiple capability types: `TEXT_GENERATION`, `IMAGE_GENERATION`, `SPEECH_TO_TEXT`, `TEXT_TO_SPEECH`, `EMBEDDINGS`, and `CUSTOM`, each with capability-specific metadata schemas
- [ ] Miners register per-capability and the orchestrator routes requests to miners whose declared capabilities match the requested `modelId` capability type
- [ ] Pricing for non-LLM tasks is expressed in appropriate units: per image (image gen), per second of audio (speech), per embedding vector (embeddings) — and the SDK abstracts unit conversion from ROKO
- [ ] Result artifacts for non-text outputs reference the binary content via IPFS CID with a declared MIME type; the SDK provides a `fetchBinaryResult(requestId)` helper that streams the content
- [ ] The marketplace UI and API documentation clearly categorize available models by capability type with appropriate filtering

## Notes
Multi-modality is a significant expansion of protocol scope but essential for market relevance. Phase 1 may launch with LLM-only support; this story represents a planned extension. The pricing unit abstraction (criterion 3) is important for consumer UX — developers should not need to know the internal ROKO-per-token conversion for image jobs.
