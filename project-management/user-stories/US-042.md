# US-042: Set a Maximum Fee Cap per Inference Request

**As a** app developer/API consumer,
**I want to** specify a maximum ROKO fee I am willing to pay per inference request,
**So that** my application never spends more than I budget for a single call, even during demand spikes.

## Acceptance Criteria
- [ ] The `submitRequest` function accepts a `maxFee` parameter; the contract reverts if the current market clearing price exceeds `maxFee` at submission time
- [ ] If a miner completes the job for less than `maxFee`, the difference is returned to the requester's address in the same settlement transaction
- [ ] The SDK exposes a `getEstimatedFee(modelId, tokenCount)` helper so developers can set `maxFee` programmatically with a configurable headroom multiplier (e.g., `1.1x`)
- [ ] Fee cap validation is a pre-condition check that fails fast before any token transfer, minimizing wasted gas on doomed transactions
- [ ] Documentation and SDK type definitions clearly distinguish between `maxFee` (hard cap, transaction-level) and `estimatedFee` (advisory, off-chain)

## Notes
Without a fee cap, application costs become unpredictable in high-demand periods. The refund mechanism (criterion 2) requires the contract to track `maxFee` vs. actual settled fee per `requestId`.
