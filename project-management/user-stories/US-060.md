# US-060: Encrypt Inference Inputs So Miners Can't See Raw Data

**As a** enterprise architect,
**I want to** encrypt my inference prompt before it leaves my application so that miners process it without reading its contents,
**So that** sensitive business data (contracts, PII, proprietary information) is never exposed to third-party compute providers.

## Acceptance Criteria
- [ ] The SDK provides a `confidentialSubmit` method that encrypts the prompt payload using the assigned miner's public key before transmission; the miner decrypts inside a Trusted Execution Environment (TEE) and the plaintext never touches an unprotected memory space
- [ ] Miners advertising TEE capability (e.g., Intel TDX, AMD SEV) register a TEE attestation certificate on-chain; consumers can require TEE-only routing as a submission flag
- [ ] The miner's TEE produces a remote attestation quote proving the inference ran inside a valid enclave; this quote is included in the result artifact and verifiable by the consumer
- [ ] If no TEE-capable miner is available for the requested model, the gateway returns an error rather than silently falling back to unencrypted execution when `requireTee: true` is set
- [ ] Documentation clearly describes the threat model: what is protected (prompt content), what is not (metadata such as timing, model choice, requester address), and known limitations of the TEE approach

## Notes
This is the hardest story in the marketplace domain. Full confidential computing with TEE attestation is a significant engineering investment and hardware dependency. A v1 approximation might use asymmetric encryption at rest in transit without TEE, accepting a weaker but pragmatic threat model. The threat model documentation (criterion 5) is non-negotiable regardless of implementation approach.
