# US-094: Test Flash Loan Attack Resistance on Staking Contracts

**As a** security auditor,
**I want to** execute a suite of flash loan attack simulations against the ROKO staking contracts,
**So that** I can confirm an attacker cannot use borrowed liquidity within a single transaction to manipulate staking balances, vote weights, or reward calculations.

## Acceptance Criteria
- [ ] Staking contract enforces a minimum lock period (e.g., 1 epoch / ~7 days) before staked ROKO counts toward voting weight or reward eligibility; flash-staked tokens have zero governance weight
- [ ] Vote weight snapshot uses block number at proposal creation; any tokens staked after the snapshot block carry zero weight for that proposal
- [ ] A Foundry test simulates borrowing 100% of ROKO liquidity via Aave flash loan, staking it, attempting to vote or claim rewards, and verifies all such attempts revert
- [ ] Staking rewards are calculated based on time-weighted stake, not instantaneous balance; snapshot-based reward claims are not vulnerable to same-block manipulation
- [ ] Audit report documents the flash loan resistance mechanisms with references to the relevant contract code and test cases

## Notes
Flash loan attacks on governance are a well-documented attack class (Beanstalk, Compound). The lock period is the primary defense; snapshot voting is the secondary defense. Ensure the lock period cannot be bypassed via a wrapper contract or delegate call. Test both ERC-20 and ERC-4626 vault staking interfaces if applicable.
