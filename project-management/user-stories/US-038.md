# US-038: Receive Inference Results Referenced by IPFS Hash

**As a** app developer/API consumer,
**I want to** retrieve my inference result via an IPFS content hash stored on-chain,
**So that** I have a verifiable, permanent, tamper-evident reference to the output my application received.

## Acceptance Criteria
- [ ] When a miner submits a completed result, the contract stores the IPFS CID in the `requestId` record and emits a `ComputeComplete` event containing `requestId` and `resultCid`
- [ ] Developer can poll or subscribe to `ComputeComplete` events filtered by their address or `requestId` to retrieve the CID without scanning all events
- [ ] The IPFS-pinned result payload includes: `requestId`, `modelId`, `promptHash`, `outputText` (or binary blob), `minerAddress`, `timestamp`, and a miner-signed attestation
- [ ] SDK helper resolves the CID to content automatically, with configurable IPFS gateway fallback list
- [ ] Attempting to retrieve a result for a `requestId` that is still `PENDING` or `ASSIGNED` returns a clear status indicator rather than an error

## Notes
Using IPFS CIDs keeps large inference outputs off-chain while maintaining cryptographic binding to the on-chain record. Arweave can serve as a permanent backup pin for archival needs. The miner-signed attestation in the payload is the raw material for challenge/dispute flows (see US-040).
