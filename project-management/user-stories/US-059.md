# US-059: Route Inference to Specific Geographic Regions for Latency

**As a** enterprise architect,
**I want to** specify a preferred geographic region or restrict inference routing to miners within that region,
**So that** my end-users receive consistently low-latency responses and I can comply with data residency policies.

## Acceptance Criteria
- [ ] Miners self-declare their region at registration time (e.g., `us-east`, `eu-west`, `ap-southeast`) and the on-chain miner registry includes a `region` field that governance can update via dispute if falsely declared
- [ ] The `submitRequest` call accepts an optional `preferredRegion` parameter; the orchestrator routes to eligible miners in that region first, falling back to any region only if no regional miner is available within 5 seconds
- [ ] A `strictRegion` flag (boolean) can be set to disable fallback — requests fail rather than route outside the declared region, supporting hard data residency requirements
- [ ] Region coverage and availability are displayed on the marketplace map view, showing miner density and average latency per region
- [ ] The SDK exposes `setDefaultRegion(region)` as a client-level config so all requests from that client instance inherit the region preference without per-call configuration

## Notes
Geographic routing relies on miner self-declaration, which is imperfect. Consider a latency-probe oracle that periodically measures round-trip time from known network endpoints to verify declared regions. Hard `strictRegion` enforcement (criterion 3) is important for GDPR/data-residency compliance use cases.
