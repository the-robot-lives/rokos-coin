# US-051: Submit a Training Job to the Distributed Compute Network

**As a** AI researcher,
**I want to** submit a distributed model training job to the ROKO compute network,
**So that** I can leverage decentralized GPU resources for experiments without provisioning my own cluster.

## Acceptance Criteria
- [ ] A `TrainingJobRegistry` contract accepts job submissions with: dataset CID (IPFS/Arweave), base model identifier, training config CID, max budget (ROKO), and deadline
- [ ] The off-chain orchestrator decomposes the job into shards and assigns shards to miners meeting minimum hardware requirements (e.g., H100 or A100 with NVLink for distributed training)
- [ ] Budget is escrowed at submission and incrementally released to miners as verified checkpoints are submitted; unspent budget is returned on job completion
- [ ] Researchers receive a job dashboard URL showing: assigned miners, per-shard progress, loss curve (polled from checkpoint CIDs), estimated completion, and current spend
- [ ] Job submission is rejected with a descriptive error if no miners with sufficient hardware and available capacity exist at submission time

## Notes
Distributed training is substantially more complex than inference: it requires multi-node coordination, gradient synchronization, and fault tolerance. Phase 1 may support only single-node fine-tuning jobs. The incremental payment model (criterion 3) protects consumers from paying for incomplete work. See US-052 for checkpoint monitoring.
