# US-048: Reserve Dedicated Compute Capacity for SLA Guarantees

**As a** enterprise architect,
**I want to** reserve dedicated GPU capacity from a miner with a binding SLA,
**So that** my production workloads have guaranteed throughput and latency that aren't subject to spot market fluctuations.

## Acceptance Criteria
- [ ] An on-chain `ReservationAgreement` contract can be created between a consumer and a miner, specifying: reserved throughput (req/min), latency SLA (p95 ms), duration, and ROKO rate per period
- [ ] Consumers pre-stake ROKO for the full reservation period; miners pre-stake collateral equal to a governance-defined multiple of the reservation value as SLA bond
- [ ] During the reservation period, the miner's orchestrator prioritizes reserved requests over spot requests and the on-chain scheduler routes the consumer's requests to the reserved miner
- [ ] SLA breaches (missed latency or throughput) are automatically detected by on-chain oracle attestation and trigger partial collateral release to the consumer proportional to breach severity
- [ ] Reservation agreements are transferable (ERC-721-like NFT) so enterprises can resell unused capacity on a secondary market

## Notes
This is a significant protocol extension beyond spot inference. The on-chain oracle for SLA monitoring (criterion 4) is the hardest piece — consider a committee of verifier nodes that attest to performance samples. The NFT transferability (criterion 5) is a could-have for v2.
