# US-066: Set Up a Prepaid ROKO Balance for Frictionless API Calls

**As a** app developer/API consumer,
**I want to** deposit a prepaid ROKO balance into a developer account contract from which inference fees are automatically deducted,
**So that** my application can make API calls without requiring a wallet signature and on-chain transaction for every individual request.

## Acceptance Criteria
- [ ] A `PrepaidAccount` contract stores a deposited ROKO balance per address; deposits are made via a single `deposit(amount)` call and the balance is tracked on-chain
- [ ] At request submission, the API gateway deducts the fee from the prepaid balance atomically if sufficient balance exists, without requiring a per-request ERC-20 approval
- [ ] If the prepaid balance falls below a configured minimum threshold, an email/webhook alert is sent and new requests fail with a descriptive `InsufficientPrepaidBalance` error code rather than silently routing to direct-payment mode
- [ ] Balance withdrawals require a multi-step process: initiate withdrawal → wait governance-defined delay (e.g., 24 hours) → claim — protecting against private key compromise draining the balance instantly
- [ ] The developer dashboard displays current prepaid balance, estimated requests remaining at current spend rate, and a top-up button with a suggested amount

## Notes
The prepaid model eliminates the UX friction of per-request approval flows, which is critical for server-side applications. The withdrawal delay (criterion 4) is a security feature — consider making it configurable per account with a governance-set maximum. The balance depletion warning (criterion 3) must be proactive enough to prevent unexpected service interruption.
