# US-053: Resume an Interrupted Training Job from Last Checkpoint

**As a** AI researcher,
**I want to** resume a training job that was interrupted (due to miner failure or budget pause) from the last verified checkpoint,
**So that** I don't lose training progress and can continue without restarting from epoch zero.

## Acceptance Criteria
- [ ] A `resumeJob(jobId)` contract call re-activates a job in `PAUSED` or `FAILED` status, using the latest verified checkpoint CID recorded on-chain as the starting state for the next miner assignment
- [ ] The orchestrator provisions a new miner (or re-assigns the same miner if available), passes the checkpoint CID as the initialization artifact, and transitions job status back to `RUNNING`
- [ ] Resume requires topping up escrow if the remaining budget is below a minimum threshold; the contract prompts with the required amount
- [ ] The job timeline in the dashboard records the interruption event, checkpoint used for resume, and new miner assignment with timestamps for full auditability
- [ ] Resuming a job that has already `COMPLETED` or been `CANCELLED` reverts with a clear error code

## Notes
Fault-tolerant job execution is a key differentiator for decentralized compute vs. centralized cloud. The checkpoint-based resume (criterion 2) requires the miner's training runtime to support loading from an arbitrary checkpoint CID — this should be a required capability for miners accepting training jobs.
