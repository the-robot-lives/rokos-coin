# US-052: Monitor Training Job Progress with Checkpointing

**As a** AI researcher,
**I want to** monitor my training job's progress in real time and have checkpoints persisted automatically,
**So that** I can observe loss curves, detect divergence early, and ensure my work is not lost if a miner drops out.

## Acceptance Criteria
- [ ] Miners submit checkpoint CIDs to the `TrainingJobRegistry` at governance-defined intervals (e.g., every N steps or every M minutes), and the contract records checkpoint sequence numbers with timestamps
- [ ] The researcher dashboard displays a live loss/metric curve by polling checkpoint CIDs and rendering the training log data stored in each checkpoint artifact
- [ ] Each checkpoint artifact stored on IPFS includes: step number, loss value, optimizer state, model weights delta (or full weights), and miner signature
- [ ] If a miner fails to submit a checkpoint within the expected window, the orchestrator detects the miss, reassigns the shard to a replacement miner starting from the last valid checkpoint, and emits a `ShardReassigned` event
- [ ] Researchers can manually trigger a checkpoint submission at any time via the SDK `requestCheckpoint(jobId)` call

## Notes
Checkpointing frequency is a tradeoff between data safety and IPFS storage cost/overhead. A governance parameter controlling minimum checkpoint interval prevents miners from skipping checkpoints to save time. The manual checkpoint trigger (criterion 5) is important for researchers who want to inspect state before a critical training phase.
