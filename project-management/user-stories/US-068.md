# US-068: Create a Governance Proposal with On-Chain Voting

**As a** Governance Council member,
**I want to** create and submit an on-chain governance proposal with a defined voting period,
**So that** the DAO can formally deliberate and vote on protocol changes in a transparent, tamper-proof manner.

## Acceptance Criteria
- [ ] Any address holding a minimum proposal threshold of staked ROKO (configurable, default 10,000 ROKO) can submit a proposal with a title, description, and executable calldata
- [ ] Proposal is assigned a unique on-chain ID and enters a configurable voting delay period (e.g., 48 hours) before voting opens
- [ ] Voting period is configurable per proposal category (e.g., 5 days for parameter changes, 14 days for upgrades)
- [ ] Proposal state machine transitions correctly: Pending → Active → Succeeded/Defeated → Queued → Executed/Expired
- [ ] Proposal creation emits a `ProposalCreated` event indexed by ID, proposer, and start/end blocks

## Notes
Follows OpenZeppelin Governor pattern. Proposal calldata should be validated for target contract allowlist to prevent arbitrary execution. Consider snapshot block for vote weight to prevent flash-loan manipulation.
