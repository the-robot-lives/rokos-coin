# US-076: Veto a Malicious Proposal During Timelock Window

**As a** security auditor,
**I want to** trigger an emergency veto of a queued governance proposal that contains malicious or erroneous calldata before it executes,
**So that** the protocol has a last-resort defense against governance attacks that slip through the voting process.

## Acceptance Criteria
- [ ] A designated Guardian multisig (e.g., 4-of-7 security council) holds veto authority via a `cancel()` function on the TimelockController
- [ ] Veto can only cancel queued proposals; it cannot affect active votes, revert past executions, or freeze the protocol
- [ ] Veto action emits a `ProposalCancelled` event with the guardian address, proposal ID, and a required on-chain reason hash (IPFS CID to full justification)
- [ ] Guardian veto power itself is subject to governance: the guardian multisig composition can be changed via a ProtocolUpgrade proposal
- [ ] dApp surfaces a visible "Queued — under security review" banner for proposals in timelock, with a countdown and a link to submit veto concerns to the guardian

## Notes
The veto mechanism must not become a centralization risk: guardian should be a well-distributed multisig and its use should be extremely rare and publicly justified. Consider a "guardian sunset" clause that requires periodic renewal via governance vote.
